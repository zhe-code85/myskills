# Module Plan Template

用于在 `rtl-coding` 输入准备度通过后、写 RTL 前，产出 `<module_name>/module_plan.md`。它是本模块 RTL 的编码 plan：先想清楚怎么实现，落成文档供人评审，批准后作为分段编码的依据和 review 的回对基准。

## 定位

- `module_plan.md` 是本技能的产出，不是上游输入。它只规划实现层 how（模块内部结构、控制路径、数据路径），不定义接口/边界/对外行为等 what。
- 目标有二：写码前给人评审实现思路，尽早拦下错误设计；写码后可追溯——plan 每条决策都能对应到 RTL 的具体信号或逻辑块。
- 是 plan 不是设计规格：够指导编码、够给人评审即可，不写成面面俱到的说明文。

## 编写原则

- 只从已确认的模块设计输入、指定文档和准备度结论提炼实现方案；不新增 what 决策。
- 每条实现决策写清「怎么做 + 依据」，依据回指某条已确认输入或某条编码约束。
- plan 里出现的信号名、状态名、功能块名，应与后续 RTL 命名一致，便于逐条回对。
- 碰到未闭合的 what（接口、边界、对外行为、时序契约），停止写 plan，回退准备度检查，不在 plan 里就地拍。
- 某类内容 not-applicable（如无状态机、无寄存器）时，写明 not-applicable，不强行编造。

## 模板

```markdown
# <module_name> module plan

## 1. 宏观实现结构
- 模块职责一句话概述（回指模块设计输入）。
- 内部功能块清单：每块的职责、输入依赖、输出结果。
- 控制路径与数据路径如何划分，块间关键信号。
- 主数据流与主控制流走向（可用文字框图描述）。

## 2. 控制路径怎么做
- FSM：状态清单、复位初态、状态转移条件（顶层 case 以现态寄存器为选择条件）。
- 握手/背压：valid/ready 或等价协议的产生与阻塞条件。
- 仲裁/使能：多源竞争如何裁决，各使能信号的产生条件。
- 异常/边界控制：空/满、非法输入、错误检测与上报、复位后重启。

## 3. 数据路径怎么做
- 数据选择与运算：主要 mux、算子、位宽处理。
- pipeline：分几级、各级寄存器、级间信号、时序意图（吞吐/延迟）。
- buffer/存储：结构意图（FIFO/RAM/寄存器组）、深度、读写口。
- 输出绑定：每个 output 端口对应的 reg 信号与时序更新块。

## 4. 关键决策与依据
- 决策：<实现层选择> —— 依据：<回指的输入约束或编码约束>。
- CBB/IP：若准备度要求复用，记录被复用模块路径与例化意图；未使用写 not-applicable。
- 已知风险/不确定点：留给编码和验证关注的实现敏感点。
```

## 完成标准

- 能据 plan 画出模块内部框图：功能块 + 块间信号 + 主数据流/控制流。
- 每个功能块职责清晰，且能回指某条已确认的模块设计输入。
- 控制路径、数据路径写到「照 plan 能落 RTL」的粒度，而非泛泛描述。
- 无未闭合的 what 决策残留在 plan 中。

## 示例写法

- Good: `S_BUSY 态下 valid_o 拉高，仅当 ready_i 未背压时推进到 S_DONE —— 依据：接口约定 valid/ready 握手。`
- Good: `数据路径 3 级 pipeline，stage1 取数、stage2 运算、stage3 输出寄存 —— 依据：吞吐约束每拍一笔、延迟允许 3 拍。`
- Good: `复用 rtl/common/sync_fifo.v 作输入缓冲，深度 16 —— 依据：准备度指定 CBB/IP 路径。`
- Bad: `实现控制逻辑`。
- Bad: `按需求做数据通路`。
- Bad: `把接口改成 AXI-Lite`（这是 what 决策，应回退准备度，不在 plan 里拍）。
