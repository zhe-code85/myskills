# Design Plan Template

用于在 `rtl-coding` 输入准备度通过后、写 RTL 前生成 Design Plan。Design Plan 固定本模块的编码基线：需求、接口、时序、微架构、行为规则和验证目标。通用 RTL 编码规范、可综合规则、lint 策略由专门技能处理。Design Plan 落盘为 `<module_name>/module_plan.md`，输出后应等待人工评审和 Agent 确认，再进入 RTL 编码。

## 1. 需求摘要

将用户需求整理成清晰设计目标，不要直接复制原话。

必须说明：
- 模块实现什么功能。
- 主要输入和输出是什么。
- 核心行为是什么。
- 哪些功能不属于本次需求。

要求：
- 不加入用户未要求的扩展功能。
- 需求不清时，不直接开始设计。
- 模糊点放入“待确认问题”。

## 2. 假设与待确认问题

列出继续设计所依赖的假设，以及需要用户确认的问题。Agent 不得把模糊需求当成确定事实。

### 假设

列出当前临时采用的设计假设，例如时钟数量、复位类型和有效电平、输入输出是否寄存、数据延迟、参数默认值、协议默认行为。

### 待确认问题

列出需求缺失或有歧义的问题，例如读数据延迟、valid/ready 反压、满空边界、同周期读写、跨时钟域、错误状态或异常输出。

### 需要用户确认的事项

单独列出会影响接口、时序、功能行为、验证结果的问题。未确认前，不应作为最终设计约束。

## 3. 设计范围

说明本模块负责什么、不负责什么，防止设计范围膨胀。

### 范围内

列出本模块必须实现的功能，例如数据缓存、状态控制、数据通路处理、指针或计数管理、valid/ready 握手、状态标志输出。

### 范围外

列出本次不实现的功能，例如总线封装、CDC 支持、ECC/parity、低功耗门控、后端时序优化、系统级集成逻辑。

## 4. 顶层接口

定义模块对外接口，作为后续 RTL、testbench、文档生成的接口契约。

### 参数

列出所有参数，包括默认值、取值限制、作用说明。

| 参数名 | 默认值 | 取值限制 | 说明 |
|---|---:|---|---|
| DATA_WIDTH | 32 | >= 1 | 数据位宽 |
| DEPTH | 16 | >= 1 | 缓存深度 |

要求：
- 参数名、默认值、限制条件必须明确。
- 参数是否影响位宽、深度、状态数，应写清楚。
- 非法参数行为不明确时，列为待确认问题。

### 端口

列出所有端口，包括方向、位宽、含义、时序属性。

| 信号名 | 方向 | 位宽 | 说明 |
|---|---|---:|---|
| clk | input | 1 | 主时钟 |
| rst_n | input | 1 | 低有效复位 |
| in_valid | input | 1 | 输入数据有效 |
| in_ready | output | 1 | 模块可接收输入 |

要求：
- 不遗漏端口方向。
- 不使用模糊位宽。
- 每个端口说明用途。
- 控制信号和数据信号分清。

### 接口协议

描述端口之间的行为关系，包括数据何时被采样、输出何时有效、valid/ready 握手规则、backpressure 行为、输入/输出数据保持稳定的条件。

### 寄存器语义

固化准备度已确认的软件可见寄存器（配置/状态/中断寄存器），作为对外契约的一部分。这里描述软件看到的行为，不描述内部实现寄存器（内部寄存器在 §6 微架构里列）。无软件可见寄存器时写“本模块无软件可见寄存器”。

| 寄存器 | 偏移 | 位域 | 读写属性 | 复位值 | 说明 / side effect |
|---|---|---|---|---:|---|
| CTRL | 0x00 | [0] enable | RW | 0 | 置 1 启动，清 0 停机 |
| STATUS | 0x04 | [0] busy | RO | 0 | 处理中回读为 1 |

要求：
- 只固化准备度已确认的寄存器语义并标注来源；语义不明确时回退准备度检查，不在此就地拍板。
- 复位值、读写属性、side effect、状态回读、中断或错误语义必须写清楚。

## 5. 时钟、复位与时序

说明模块的时钟域、复位方式和关键时序行为。这里只描述当前设计特有时序，不展开通用 RTL 编码规则。

### 时钟域

说明是单时钟还是多时钟。多时钟设计要列出每个时钟的作用，并在 CDC 章节说明跨域信号。

### 复位策略

说明复位信号名称、有效电平、复位后状态机状态、计数器状态、标志状态、valid 输出状态，以及是否存在不复位寄存器。

### 延迟

说明关键输入到输出的延迟，例如输入握手到输出、读请求到读数据、状态标志更新、done/valid/ready 的时序关系。

### 吞吐率

说明理想情况下每周期能接收几个输入、输出几个结果，pipeline 填满后的吞吐率，以及是否存在等待周期。

### 时序假设

列出外部时序前提，例如输入已同步、上游握手前保持数据稳定、下游按协议接收、配置信号在启动前稳定。

## 6. 微架构

描述模块内部如何实现需求。不写代码，但要让后续编码有明确结构。

### 内部子模块

列出内部逻辑块及职责，例如输入握手控制、数据缓存、指针管理、计数器管理、状态机控制、输出寄存器、标志生成逻辑。

### CBB/IP 复用

固化准备度已确认的复用决定：本模块用哪些已有 CBB/IP、各自例化在哪个内部子模块、可定位的模块路径；以及哪些结构确定自实现。无复用时写“本模块不复用 CBB/IP”。

| 复用结构 | CBB/IP 模块 | 路径 | 用在哪 |
|---|---|---|---|
| 同步 FIFO | sync_fifo | rtl/common/sync_fifo.v | 数据缓存子模块 |

要求：
- 只固化准备度已确认的复用模块与路径；准备度要求复用时，RTL 必须实际例化该路径对应模块，不得改自实现或替换未确认模块。
- 复用与否未确认时回退准备度检查，不在此就地决定。

### 数据通路

说明数据在哪里采样、缓存、选择、计算、转发、更新到输出，以及是否存在 bypass、pipeline 或 buffer。

### 控制通路

说明握手成功条件、计数器更新条件、状态跳转条件、读写使能条件、输出 valid 产生条件、ready/stall/busy 产生条件。

### 寄存器 / 状态元素

列出关键内部寄存器、计数器、状态变量（内部实现用，非软件可见；软件可见寄存器在 §4 寄存器语义里固化）。

| 名称 | 类型 | 复位值 | 作用 |
|---|---|---:|---|
| state_q | 状态寄存器 | IDLE | 当前状态 |
| count_q | 计数器 | 0 | 有效数据数量 |
| data_q | 数据寄存器 | 0 | 缓存输出数据 |

要求：关键状态元素必须列出；每个状态元素说明用途；复位值不明确时标为待确认。

### 关键信号

列出后续 RTL 中需要重点实现和验证的内部信号，例如 `wr_fire`、`rd_fire`、`next_state`、`count_next`、`full_next`、`empty_next`、`output_valid_next`。

## 7. 状态机 / 控制流程

如果设计包含 FSM，必须写状态定义和状态转移。如果没有 FSM，说明“本设计不需要显式状态机”。

### 状态定义

| 状态 | 含义 |
|---|---|
| IDLE | 等待启动 |
| LOAD | 加载输入或配置 |
| RUN | 正常处理 |
| DONE | 处理完成 |

### 状态转移表

| 当前状态 | 条件 | 下一状态 | 说明 |
|---|---|---|---|
| IDLE | start=1 | LOAD | 收到启动请求 |
| LOAD | load_done=1 | RUN | 加载完成 |
| RUN | done=1 | DONE | 处理完成 |
| DONE | clear=1 | IDLE | 清除完成标志 |

要求：每个状态有明确退出条件；默认保持条件要说明；reset 后状态要说明；状态转移优先级要说明。

### 非法状态处理

说明进入未定义状态时的处理方式，例如回到 IDLE、触发 error，或列为需要用户确认。

## 8. 行为规则

用明确规则定义模块行为，直接指导 reference model 和 testbench。

### 正常行为

说明标准条件下模块如何工作，例如输入握手成功时采样输入数据、输出有效时保持输出数据稳定、处理完成后拉高 done、下游未 ready 时保持输出不变。

### 边界条件

说明边界状态下的行为，例如空状态读请求、满状态写请求、同周期读写、计数器达到最大值、最小参数配置。

### 错误 / 忽略条件

说明非法输入或无效操作如何处理，例如 full 时写请求、empty 时读请求、非法配置、互斥控制信号同时出现时的优先级。

### 角落场景

列出必须验证的 corner case，例如复位过程中输入有效、连续 back-to-back 传输、满空边界切换、同周期读写、持续反压、最小/最大参数配置、状态机快速切换。

## 9. CDC / 复位域考虑

说明是否存在跨时钟域或复位域问题。如果没有，也要明确写“无 CDC”。

应说明：
- 是否有多个时钟域。
- 哪些信号跨域。
- 跨域信号方向。
- 是否需要同步器。
- 是否需要 Gray code 指针。
- 复位是否跨域释放。
- 是否需要 CDC 检查。

要求：
- 单时钟设计必须说明输入信号是否假设已同步。
- 多时钟设计不得省略 CDC 方案。
- CDC 不明确时必须列为待确认问题。

## 10. 验证敏感点交接

本节不写完整验证计划（定向/随机测试、参考模型、覆盖率、pass/fail 由 `rtl-verification` 负责）。这里只提炼本模块需要交给验证的关注点，编码后落成 `<module_name>/verification_input.md`。

列出以下几类，来源是本 Design Plan 的行为规则（§8）和实现决策，不从 RTL 反推：

- 主功能路径：正常条件下必须成立的核心行为。
- 边界与异常：空/满、溢出/下溢、非法输入、错误上报与恢复、复位后重启。
- 实现敏感点：本次实现里最容易出错、最需要验证盯住的地方，例如 pipeline 对齐、背压下的数据保持、同周期读写、FSM 快速切换、跨域握手。
- 关键约束：从行为规则和寄存器语义提炼、必须在 RTL 中可定位的断言级约束，例如 full/empty 不同时为 1、计数不超过 DEPTH、未握手时输出保持稳定。
- 建议优先级：上述关注点里应优先验证的项。

每条关注点应能在当前 RTL 中定位到具体信号或逻辑块；写成可检查的行为，避免“功能验证”这类泛泛表述。
